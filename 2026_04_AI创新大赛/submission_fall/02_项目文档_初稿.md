## 1 项目概况

### 1.1 背景和基础

在居家养老、医疗照护、社区看护等场景中,跌倒事件具有突发性强、处置窗口短、误报成本高的特点。赛题要求构建端到端纯视觉方案,在低算力端侧平台上对普通/红外视频进行实时跌倒识别,不能依赖深度图、穿戴设备或云端实时推理。

本项目围绕“高召回、低误报、低参数、低延时”展开。已有工作包括:

- 解析赛题七约束: 模型总参数不超过 20M、fp32 大小不超过 80MB、端侧推理耗时不超过 100ms、NPU 存储占用不超过 20MB。
- 完成可运行 baseline: 合成跌倒/非跌倒视频片段、纯视觉人体区域特征、tiny 时序模型、报警后处理、P@R90/P@R95 指标和延时/参数评分代理。
- 建立公开视频数据入口: 支持 OmniFall、Kaggle Fall Video Dataset 等本地视频目录生成 clip-level manifest,为后续真实数据评测做准备。
- 形成可视化产物: 分数分布图、PR 曲线、跌倒报警示例拼图,用于文档和视频展示。

当前 baseline 仅用于工程闭环验证,公开/官方测试成绩需要接入真实视频后重新评估。

### 1.2 场景和价值

项目适用于家庭摄像头、养老机构护理站、病房看护、社区独居老人安全监测等场景。相比云端视频分析,端侧闭环方案能够减少隐私视频外传,降低网络延迟,并在弱网或断网时保持本地报警能力。

与常规高复杂度视频动作识别方案相比,本项目强调轻量化和可解释性: 先用低成本视觉模块获得人体区域和姿态变化,再用短时序模型判断“快速下坠、姿态横置、持续静止”等跌倒语义,避免把快速坐下、弯腰捡物、躺下休息等日常动作简单判为跌倒。

### 1.3 所需支持

- 数据支持: 下载或申请 OmniFall、Kaggle Fall Video Dataset、赛题平台参考数据,覆盖普通光、弱光、红外、遮挡、视角变化和多人干扰场景。
- 硬件支持: 至少一类端侧验证平台,优先 RK3588/RKNN、海思 NPU 或等价低算力设备。
- 算力支持: 训练阶段可使用通用 GPU 或 CPU;推理阶段需补充端侧实测的模型大小、内存和延时。
- 评测支持: 需要统一的 clip-level 标注格式,包含跌倒事件起止、视频路径、场景标签和数据来源。

## 2 项目规划

### 2.1 整体目标

参赛期间完成一个可提交、可复现、可端侧部署验证的纯视觉实时跌倒检测系统:

- 输入: 1080P 以上相机视频,算法内部可自定义缩放分辨率和帧采样策略。
- 输出: 每个视频片段的跌倒分数、报警起止帧、clip-level TP/FP/FN 统计和可视化报告。
- 精度目标: 在召回率 90% 和 95% 两个工作点尽量提升精准率,重点降低坐下、弯腰、躺下等误报。
- 效率目标: 总参数量控制在 20M 内,端侧端到端推理耗时控制在 100ms 内,NPU 存储占用控制在 20MB 内。
- 交付目标: 项目文档、项目视频、代码、模型、使用说明、公开数据评测报告和匿名化提交包。

### 2.2 技术创新点

1. 端侧友好的跌倒语义拆解  
将跌倒从单帧分类问题拆成“人体区域变化、垂直位移、横向姿态、持续低位状态”的短时序语义组合,用低维特征表达复杂动作,降低小模型对大规模 3D-CNN/Video Transformer 的依赖。

2. tiny 时序头 + 报警状态机  
当前 baseline 使用 8 维时序统计特征和 9 参数 logistic head 输出跌倒分数,再通过报警状态机控制持续帧和冷却窗口。后续可替换为轻量 TCN/GRU,但保持低参数、可解释和易部署。

3. 长尾场景增强入口  
合成数据模块已覆盖红外灰度、噪声、遮挡、弱光代理等扰动。后续在真实数据上加入光照扰动、随机遮挡、视角仿射、运动模糊和低码率压缩增强,提高夜间红外和遮挡场景泛化。

4. 评测与部署一体化  
报告同时输出 P@R90/P@R95、参数量、fp32 大小、平均/95 分位延时和竞赛分代理,让算法改动能直接对齐赛题评分项。

## 3 实施方案

### 3.1 技术可行性分析

数据方面,赛题给出 OmniFall 和 Kaggle Fall Video Dataset 作为参考数据源。项目已实现本地视频目录扫描和 label 推断脚本,可将公开数据转成统一 manifest 后评测。对于红外、遮挡、夜间等长尾样本,计划采用公开数据增强、少量平台参考数据调参和合成扰动补充。

算法方面,跌倒检测具有明显时序语义,不必完全依赖高复杂度视频网络。轻量人体检测或姿态估计模型负责空间定位,tiny 时序模型负责行为判别,整体参数和延时更容易满足端侧约束。当前 baseline 已在无外部数据情况下跑通完整链路,说明工程结构可行。

硬件方面,方案优先选择端侧成熟部署路径: ONNX 作为中间格式,RKNN/NCNN/海思工具链作为部署目标。推理阶段将量化到 INT8 或 FP16,并对输入分辨率、帧率、检测间隔和时序窗口做联合优化。

风险主要来自真实数据分布差异、红外场景少样本、遮挡导致的人体区域不稳定和端侧 NPU 算子兼容性。项目通过 manifest 数据入口、可解释状态机和模块化模型导出降低这些风险。

### 3.2 技术细节

#### 3.2.1 系统流程

```text
1080P 视频流
  -> 抽帧/缩放/颜色空间适配
  -> 轻量人体区域提取或姿态估计
  -> 帧级几何特征: bbox 中心、宽高比、面积、可见度、速度
  -> 短时序特征聚合: 最大横置程度、下坠速度、总下移、最终高度
  -> tiny 时序分类头/状态机
  -> 跌倒分数、报警起止帧、clip-level 指标
```

当前工程实现位于 `src/`:

- `fall_synth.py`: 生成 fall、walk、sit、bend、lie 五类合成片段。
- `fall_features.py`: 从纯视觉帧中提取前景人体框和时序几何特征。
- `fall_detector.py`: 根据下坠、横置和持续状态输出报警窗口。
- `fall_tiny_model.py`: 8 维输入、9 参数 z-score logistic 时序头。
- `run_fall_pipeline.py`: 端到端评测入口,输出 JSON 报告和分数文件。
- `viz_fall_report.py`: 生成分数分布、PR 曲线和报警示例图。

#### 3.2.2 当前 baseline 结果

最近一次 synthetic proxy 评测:

- 评测片段: 100 个合成 clip,其中 fall 20 个、non-fall 80 个。
- 输入尺寸: 160 x 120,每个 clip 32 帧。
- 当前 scorer: tiny model。
- tiny head 参数量: 9 个参数;报告中完整系统暂按 0.001M 代理参数估计。
- P@R90: 1.0。
- P@R95: 1.0。
- 平均单帧后处理耗时: 约 0.35ms。
- 95 分位单帧后处理耗时: 约 0.37ms。

说明: 上述结果来自合成代理数据,只能作为工程闭环和指标计算验证。正式报告必须补充公开视频和赛题数据上的真实成绩,且端到端耗时必须包含人体检测/姿态模型。

#### 3.2.3 指标设计

赛题核心评价为:

- 精度分: 召回率达到 90% 和 95% 时的平均精准率,权重 50%。
- 参数量分: 参数越少得分越高,基线约束为 20M,超过 40M 为 0 分。
- 推理耗时分: 延时越低得分越高,基线约束为 100ms,超过 200ms 为 0 分。

项目报告以 `fall_pipeline_report.json` 记录这些字段,保证每次模型更新都能追踪对竞赛得分的影响。

#### 3.2.4 后续模型升级

baseline 的前景人体框适合验证流程,正式方案建议升级为:

- 人体检测: YOLOv8n/YOLOv10n/NanoDet/MobileNet-SSD 等轻量检测器,导出 ONNX 后量化部署。
- 姿态或关键点: 在端侧资源允许时加入轻量关键点模型;若资源不足,保留 bbox 几何时序方案。
- 时序判别: 从当前 logistic head 升级为 tiny TCN 或 GRU,输入 16-32 帧低维特征,控制参数量在几十 K 到几百 K。
- 报警后处理: 保留多帧确认、误报冷却、低置信过滤和场景自适应阈值。

#### 3.2.5 端侧部署计划

1. 以 ONNX 导出检测/姿态模型和时序头。
2. 使用 RKNN/NCNN/海思工具链做 FP16/INT8 量化。
3. 测试 1080P 输入下的整体链路: 解码、缩放、检测、时序判别、报警输出。
4. 记录参数量、fp32 大小、量化模型大小、NPU 内存、平均延时和 95 分位延时。
5. 将实测结果写入项目文档和运行说明。
